线程传参、detach的陷阱等

传递临时对象作为线程的参数

1.要避免的陷阱1

如果把 main主函数中的 join 换成 detach： mytobj.detach();

程序可能就会出现问题了，根据调试观察，函数myprint中的形参i的地址和原来main主函数中的mvar地址不同(虽然形参是引用类型)，这个应该安全(也就是说thread类中的构造函数实际是复制了这个参数)，而函数myprint中形参pmybuf指向的内存铁定是main中mybuf的内存，这段内存是主线程中分配的，所以一旦主线程退出，子线程再使用这块内存肯定是不安全的。

所以，使用了detach这种方式创建线程，记住不要往线程里面传递引用、指针之类的参数。

如何修改上述带么呢？把第二个形参类型改为 const string& 试试。

main主函数中会将mybuf这个char数组隐式构造成string对象(myprint函数的第二个参数)。string里保存字符串地址和mybuf里的字符串地址通过断点调试，发现确实是不同。改造后感觉是线程安全了，但真是这样么？继续往下看

2.要避免的陷阱2

上述改造完后的代码，还好是有隐藏bug的。 看看这句代码 std::thread mytobj(myprint, mvar, mybuf); 这行代码的本意是：希望系统帮助我们把mybuf隐式转换成string，这样就可以再线程中使用string，线程中就不会引用main中mybuf所指向的内存，那么，mybuf内存的销毁(回收)就跟线程没有什么关系。

但是，问题是，mybuf是在什么时候转换成string？ 如果main函数都执行完了退出了，才把mybuf往string转，那就危险了，这种情况确实存在。

最终修改：std::thread mytobj(myprint, mvar, string(mybuf)); 这里，直接将mybuf转成string对象，可以保证在线程myprint中所用的pmybuf肯定是有效的。

继续剖析：上面代码转换了之后，确实是没有问题了。继续测试，为啥生成了临时对象就没有问题？

3.总结

1.如果传递int这种简单类型参数，建议都使用值传递，不要使用引用类型，以免节外生枝。

2.如果传递类对象作为参数，则避免隐式类型转换(例如把一个char*转成string等)，全部都在创建线程这一行就构建出临时对象来，然后线程入口函数的形参位置要使用引用来作为形参(如果不使用引用可能在某种情况下会导致多构造一次临时对象，不但浪费，还会造成潜在的问题)。这样做的目的无非就是想办法避免主线程退出导致子线程的非法引用。

3.建议少用detach，甚至不使用detach，这样就不存在局部变量失效导致线程对内存非法引用的问题。

2.临时对象构造实际抓捕

从结果分析，因为是通过mvar让系统通过类A的类型转换构造函数生成myprint2需要的pmybuf对象，所以可以清楚的看到，pmybuf对象在构造的时候，threadid值为9676，而9676是所创建的线程(子线程)id，也就是这个对象居然是在子线程中构造的。如果 代码不是mytobj.join();,而是mytobj.detach();的话，那就危险了，可能主线程已经退出了，子线程才拿mvar来构造类A对象。

数据共享问题

1.只读的数据

一段共享数据，如有一个容器，如果每个数据都是只读 的，每个线程都去读，那无所谓，每个线程读到的内容肯定是一样的。

例如：有一个全局的容器 vector<int> g_v = {1,2,3};

main函数中有多个线程，每个线程都打印容器g_v中的元素值，则是安全的的，虽然结果比较乱，但是程序是稳定的、正常的。

2.有读有写

如果有线程改变了容器里面的元素等，这就会出现问题了。就会出现线程安全问题。

最简单的方法：读取的时候是不能写数据、不能做修改的，写的时候就不能读，多个线程之间也不能同时写，多个线程也不能同时读。

3.std::lock函数模板

std::lock函数模板能一次锁住两个或者两个以上的互斥量(互斥量最少2个)，它不存在多个线程因为锁的顺序问题导致死锁的风险。

std::lock的工作步骤：例如它先锁住第一个互斥量，成功加锁，然后再去锁第二个互斥量，如果加锁失败，则会把成功锁住的互斥量解锁，同时等在那里，等着所有互斥量都能锁定。所以std::lock的特点就是：要么全部成功锁定，要么都没有锁住，卡在std::lock那里不断地尝试加锁。

2)std::try_to_lock

可作为unique_lock的第二个参数。意义是：系统会尝试用mutex的lock去锁定这个mutex，但如果没有锁成功，也会立即返回，并不会阻塞在那里(使用std::try_to_lock的前提是程序员不能自己先去lock这个mutex，因为std::try_to_lock会尝试去lock，如果程序员先lock了一次，那这里就等于再次lock了，两次lock的结果就是程序卡死了)。

当然，如果lock了，在离开unique_lock作用域或者从函数返回时会自动unlock。

3)std::defer_lock

unique_lock所支持的另一个第二参数：std::defer_lock（用这个defer_lock的前提是程序员不能自己先去lock这个mutex，否则报异常）。

std::defer_lock的意思就是初始化这个mutex，但是这个选项表示并没有给这个mutex加锁，初始化了一个没有加锁的mutex。

unique_lock的成员函数

1.lock

给互斥量加锁，如果无法加锁，会阻塞一直等待拿到锁。

2.unlock

针对加锁的互斥量，给该互斥量解锁，不可以针对没加锁的互斥量使用，否则报异常。

在加锁互斥量后，随时可以使用该成员函数再重新解锁这个互斥量，当然，解锁后，若有需要，还可以再重新加锁。

所以，unique_lock因为lock与unlock成员函数，所以可以随时加解锁。

3.try_lock

尝试给互斥量加锁，如果拿不到锁，则返回false，如果拿到了锁，则返回true。这个成员函数不阻塞。

4.release

返回它所管理的mutex对象指针，并释放所有权。也就是这个unique_lock和mutex不再有关系了。注意严格区别release和unlock这两个成员函数的区别，unlock只是让该unique_lock所管理的mutex解锁而不是解除两个的关联关系。

一旦解除该unique_lock和所管理的mutex的关联关系，如果原来mutex对象处于加锁状态，则程序员有责任负责解锁。

总结：lock了就需要unlock。使用了unique_lock并对互斥量lock之后，可以随时unlock。当需要访问共享数据时，可以随时lock，访问完毕后随时unlock，即便忘记了unlock，在离开其作用域时，也会自动unlock。

std::call_once

c++11引入的函数，这个函数的第二个参数是某个其他的函数名。

call_once的功能就是能够保证函数只被调用一次。即便在多线程下，也能保证这个函数只被调用一次。

所以从这个角度讲，call_once也是具备互斥量的能力的，而且从效率上据说比互斥量消耗的资源更少。

引入 std::once_flag这个结构，理解为一个标记即可。call_once就是通过这个标记来决定对应的函数是否执行。调用call_once成功后，call_once会反转这个标记的状态，这样再次调用call_once后，对应的函数就不会再次被执行了。
